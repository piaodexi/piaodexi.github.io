# chap05 mvc 아키텍처

# MVC란..
- 핵심은 클라이언트의 요청을 3가지 역활자가 나누어 처리하는 것 
사용자와의 상호작용은 VIEW컴포넌트가 담당하고, 데이터의 처리는 MODEL 컨포넌트, CONTROLLER는 뷰와 모델을 제어하여 업무 처리를 수행

- 이렇게 역활을 나누어 작업을 처리하는 방식이 객체지향 프로그래밍의 특징  
(가능한 역활을 작은 단위로 나누면 다른 프로젝트에서 재사용할 가능성 올라감)

# MVC구조에서 VIEW 컴포넌트를 정의하는 방법

- 기존에 서블릿이 하던 작업 중에서 화면 출력과 관련된 기능을 분리하여 JSP페이지로 정의  
보다 쉽게 HTML화면을 만들고 클라이언트로 출력하기 위해 등장한 기술이 JSP

# JSP 주요 특징 
- JSP 페이지는 출력문을 표현하는 **탬플릿 데이터와 지시자<%@ ...%>,선언문<%! ...%>,스크립트릿<%...%>,표현식<%=...%>** 등으로 구성됨.  
- JSP 페이지는 그 자체로 실행되지 않고 서블릿을 생성하기 위한 소스로 사용됨.
- JSP 엔진은 JSP파일을 참고하여 서블릿을 생성함,서블릿 파일을 만들 때 템플릿 데이터는 자바 출력문으로 바뀌고, 기타 JSP 전용 태그는 특정 자바코드로 바뀜

# JSP 전용 태그 중 하나인 액션(ACTION)
액션태그는 자바 객체를 생성하거나 보관소에서 값을 꺼낼 때 씀.
## 서블릿 및 JSP에서 제공하는 4가지 보관소의 생명주기
- PAGE: JSP페이지에서만 유효
- REQUEST: 요청에서 응답까지 유효 
- SESSION: 최초 요청에서 브라우저를 닫을 때까지 유효(보통 로그인해서 로그아웃할 때까지를 SESSION의 유효범위로 봄)
- APPLICATION: 웹 애플리케이션의 시작에서 종료할 때까지 유효
**객체를 공유할 경우 그 유효 범위를 따져서 적절한 보관소를 선택해야 함**

# 모델 컴포넌트
**모델 컴포넌트를 만들 위해 서블릿으로부터 데이터 처리 로직을 분리하여 DAO를 정의  
DAO를 도입하여 얻는 이점**  

- 데이터 처리 부분을 공유함으로써 중복 코드를 줄일 수 있음.
- 데이터 구조에 변경이 발생하면 그 변경 사항을 적용하기 쉽다는 것 
- 다른 프로젝트에서 재사용하기가 쉽움 

# ServletContextListener
웹 어플리케이션이 시작되거나 종료될 때 호출할 메서드를 정의한 인터페이스
- 서블릿 컨테이너는 웹 애플리케이션의 시작과 종료를 알리기 위해 이벤트를 발생시킴
**만약 이런 이벤트에 반응하여 뭔가 작업을 수행하고 싶다면, ServletContextListener의 구현체를 만들고 web.xml파일에 등록하면 됩니다.**

# Servletcontext(application)
서블릿들 사이에서 값을 공유하기 위해 위 객체를 사용함.
이 객체는 웹 애플리케이션이 시작할 때 생성되고 종료할 때까지 유지됩니다.  
따라서 웹 애플리케이션이 실행되는 동안 공유할 값이 있다면 이 객체에 보관하면 됩니다.  
해당 프로젝트에서는 dao객체를 공유하기 위해 ServletContext에 보관함.

# javax.sql 확장 패키지
- 이 패키지에서 제공하는 DataSource는 DB 커넥션 풀 기능을 제공하기 때문에 매우 편리
- DataSource는 서버에서 관리하기 때문에 데이터베이스가 바뀌거나 JDBC 드라이버가 변경되더라도 애플리케이션 쪽에는 영향을 미치지 않음.
- 톰캣 서버에 DataSource를 등록해 놓으면 , JNDI 이름으로 이서버 자원을 찾을 수 있어서 개발 편이성이 향상됨. 


# DB 커넥션 풀 
**커넥션 풀이란 DB와 미리 connection(연결)을 해놓은 객체들을 pool(웅덩이)에 저장 해두었다가, 클라이언트 요청이 오면 커넥션을 빌려주고, 볼일이 끝나면 다시 커넥션을 반납받아 pool에 저장하는 방식을 뜻함. 

클라이언트가 DB와 연결이 필요할 때 , connection pool에서 connection을 빌려와 db에 접근해서 쿼리를 막 날린 후, 볼일이 끝나면 사용했던 connection을 다시 pool에 반납을 하는 과정을 표현

## 특징/장단점
- 클라이언트가 서버로 게시글 데이터를 전달
- 비즈니스 로직 처리
- DB와 커넥션을 맺은 후 , DB에 게시글 데이터를 저장
- DB와 커넥션을 끊고, 비즈니스 로직 처리
- 클라이언트에 응답


## EX
서비스가 커져서 유저가 한명이 아니라 동시접속자 10명이라면 
위와 같은 경우 매번 db와 커넥션을 맺고 끊는 과정이 필요한데 커넥션 풀을 사용하게 되면
미리 연결을 맺고 있는 커넥션들이 있기 때문에 커넥션을 맺고 끊는 과정이 불필요
즉 , db접근시 **불필요한 작업(커넥션 생성,끊기)이 사라지므로 성능향상을 기대**할 수 있음.
